# ADR-003: Explicit Server Guards in Pages and API Handlers

## Status

Accepted

## Date

2026-05-13

## Context

The application currently renders server pages directly from repository-backed read models and exposes route handlers for board reads, order detail reads, kitchen mutations, and salão exception acknowledgement. There is no global middleware, and the existing write paths are already centralized in a small set of API handlers.

The PRD requires server-enforced authorization for protected reads and mutations. The selected technical direction favors minimal churn and explicit enforcement over a broad middleware rewrite.

## Decision

Implement area authorization through shared auth utilities that are called explicitly from protected server pages and API handlers.

The authorization matrix for this release is:

- kitchen areas may read the kitchen board
- kitchen areas may read order detail only when the focused kitchen matches the authenticated area
- kitchen areas may mutate only their own kitchen tickets and items
- salão may read only salão-specific read models and may acknowledge sync exceptions
- salão may not read kitchen board or order detail payloads

This enforcement will live in shared server-side helpers that parse the signed area cookie, validate the active session, and apply route-specific policy checks before any protected read model or mutation runs.

Catalog management is not part of the first operational area matrix. The existing `/catalog` page and `/api/catalog/*` write surfaces will join the same auth stack but deny all current Stage 6 roles. They will be removed from kitchen and salão navigation and remain deferred until a later admin or manager scope is defined.

Guard evaluation order is part of this decision. Protected pages and API handlers must resolve and authorize the active area session before any operational side effect runs, including provider-sync refresh triggers, repository mutation calls, or expensive read-model preparation. Unauthenticated or wrong-area requests must fail fast.

## Alternatives Considered

### Alternative 1: Middleware-First Gating

- **Description**: Use Next middleware as the primary access-control layer for most page routes and leave route handlers with lighter checks.
- **Pros**: Centralizes some route gating and can redirect earlier in the request flow.
- **Cons**: Still requires handler-level enforcement for API safety, adds matcher complexity, and makes fine-grained kitchen-versus-salão policy harder to reason about from the point of use.
- **Why rejected**: Explicit guards in the routes and pages that already own the read or write behavior are easier to audit and test in this codebase.

### Alternative 2: Client-Gated Reads with Server-Side Write Checks

- **Description**: Let authenticated clients read broadly and rely on UI hiding plus mutation checks for safety.
- **Pros**: Lowest change volume to current read APIs.
- **Cons**: Violates the PRD requirement for server-enforced read authorization and leaves internal payloads exposed to the wrong area.
- **Why rejected**: The first hardening slice must protect reads as well as writes.

## Consequences

### Positive

- Keeps authorization decisions close to the route or page they protect.
- Reuses the current route structure instead of introducing a parallel access-control stack.
- Makes wrong-area failures straightforward to cover in handler and page tests.

### Negative

- Multiple pages and handlers must call the shared guard helpers consistently.
- Some current shared read models will need to split so salão no longer consumes kitchen board payloads.

### Risks

- Missing a guard in one handler would create an inconsistent protection surface.
- Existing links and queries may point users toward routes their area can no longer open.
- Page-level and API-level policies could drift if they are not backed by the same helper functions.

## Implementation Notes

This decision implies:

- a dedicated salão read endpoint rather than reusing the full board payload
- canonical kitchen focus derived from the authenticated area for order detail routes
- explicit wrong-area redirects for pages and explicit `403` responses for API handlers
- protected catalog routes that are blocked for `kitchen-1`, `kitchen-2`, and `salon` in this release
- guard execution before `maybeRefreshRuntimeProviderSync()` or equivalent operational work in protected routes and pages
