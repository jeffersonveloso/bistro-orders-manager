# ADR-004: Dual-Layer Authorization Validation with Vitest and Playwright

## Status

Accepted

## Date

2026-05-13

## Context

The repository already uses Vitest for handler and service validation and Playwright for end-to-end workflow coverage. Operational Hardening introduces a safety boundary where regressions can appear both in low-level authorization rules and in user-visible role flows on shared-device screens.

The selected validation scope requires confidence in both blocked unauthorized access and preserved happy-path operation for each area.

## Decision

Extend both Vitest and Playwright coverage for the first release.

Vitest will validate the shared auth helpers, page or route policy functions, login and logout handlers, and all protected mutation or read handlers for unauthorized and authorized cases. Playwright will validate the main area flows end to end, including access entry, session persistence, wrong-area redirects, blocked read surfaces, and blocked or hidden wrong-area actions.

The test plan must also prove that unauthorized requests do not trigger protected operational side effects such as runtime provider refresh calls.

## Alternatives Considered

### Alternative 1: Vitest-Heavy Coverage with Minimal E2E

- **Description**: Focus mostly on handler and service tests and keep browser coverage to a small smoke pass.
- **Pros**: Fastest feedback loop and lower E2E maintenance.
- **Cons**: Misses shared-device navigation, redirect behavior, and UI affordance regressions that matter in this feature.
- **Why rejected**: Route and surface boundaries are part of the product behavior, not only the API contract.

### Alternative 2: Playwright-Heavy Coverage with Limited Unit or Handler Tests

- **Description**: Rely mostly on browser scenarios and test low-level auth logic indirectly.
- **Pros**: Strong confidence in full user flows.
- **Cons**: Slower failure diagnosis and weaker protection against direct handler regressions.
- **Why rejected**: The phase explicitly needs evidence that direct API misuse is blocked, not only that the browser path looks correct.

## Consequences

### Positive

- Covers both API-level abuse cases and real operator flows.
- Reuses the repository's established test harnesses rather than introducing a new framework.
- Makes release-gate metrics testable at the same abstraction levels they matter.

### Negative

- Increases total test maintenance for route protection and seeded auth scenarios.
- Requires stable seeded credential setup for E2E.

### Risks

- Test fixtures could diverge from runtime credential configuration.
- Browser tests may become brittle if access-entry copy or redirects change frequently.
- Handler tests may miss a page-only redirect bug unless E2E keeps that scenario covered.

## Implementation Notes

The E2E suite should cover at least:

- kitchen login and kitchen board access
- kitchen detail access for the matching kitchen
- blocked wrong-kitchen detail or mutation attempts
- salão login and salão acknowledgement flow
- blocked salão access to kitchen board and order detail

Handler or route tests should cover at least:

- unauthorized protected page or route access returns before `maybeRefreshRuntimeProviderSync()` runs
- wrong-area protected API access returns before repository mutation or replay side effects run
