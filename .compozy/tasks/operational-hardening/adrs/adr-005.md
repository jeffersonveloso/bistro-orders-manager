# ADR-005: Remove Catalog Administration from the First Operational Area Matrix

## Status

Accepted

## Date

2026-05-13

## Context

The current product exposes `/catalog`, `GET/POST /api/catalog/mappings`, and `POST /api/catalog/provider-pull` as public operational surfaces. Stage 6 Operational Hardening is introducing area-based access for `Kitchen 1`, `Kitchen 2`, and `Salão`, but the approved PRD does not include a manager or admin role in this first slice.

Leaving catalog management public would create a major escape hatch around the new boundaries because catalog writes can change routing behavior and replay external sync actions. Granting catalog administration to kitchen or salão roles would also violate the chosen operating model.

## Decision

Remove catalog administration from the first operational area matrix and protect it with the same session stack while denying all current Stage 6 roles.

For this release:

- `/catalog` is removed from kitchen and salão navigation
- `app/catalog/page.tsx` is protected and redirects authenticated non-authorized areas back to their canonical home surface
- `GET/POST /api/catalog/mappings` and `POST /api/catalog/provider-pull` require a session but return `403` for `kitchen-1`, `kitchen-2`, and `salon`
- the catalog surface is explicitly deferred until a later admin or manager scope is specified

## Alternatives Considered

### Alternative 1: Let Salão Manage Catalog

- **Description**: Reuse salão access for catalog maintenance because salão already handles sync exceptions.
- **Pros**: Preserves some existing mapping operations without adding a new role.
- **Cons**: Expands salão beyond monitoring and exception acknowledgement into a write-capable administrative surface that changes production routing.
- **Why rejected**: It breaks the approved product boundary for salão in this phase.

### Alternative 2: Keep Catalog Public for Now

- **Description**: Leave catalog routes unchanged and harden only kitchen and salão production flows.
- **Pros**: Lowest implementation effort and no interruption to existing catalog maintenance.
- **Cons**: Leaves a public write surface that can influence production behavior and undermines the new access boundary.
- **Why rejected**: It creates an unacceptable bypass around the phase objective.

## Consequences

### Positive

- Removes the largest remaining public write escape hatch from the protected operational surface.
- Keeps the first area-based release focused on the approved kitchen-versus-salão boundary.
- Avoids inventing a manager or admin role without product approval.

### Negative

- Catalog maintenance is temporarily unavailable to operational area users.
- Live provider pilots that depend on in-product mapping changes need a follow-on admin scope before relying on this flow.

### Risks

- Stakeholders may expect catalog maintenance to remain available after the auth rollout.
- Operators may lose a previously visible route and need communication about the deferral.
- A later admin scope will need to reconcile with this blocked surface cleanly.

## Implementation Notes

This ADR does not define the later admin surface. It only closes the current public route and records that catalog administration is deferred from the first Operational Hardening slice.
