# ADR-002: Signed Area Session Cookie with Dedicated Access Entry

## Status

Accepted

## Date

2026-05-13

## Context

The current application has no internal authentication layer. Stage 6 requires area-based access for `Kitchen 1`, `Kitchen 2`, and `Salão`, and the PRD requires that area sessions persist during a shift on shared fixed devices without forcing repeated PIN entry during routine work.

The first release also needs to avoid unnecessary schema churn. The selected product scope does not include individual accounts, PIN management UI, or manager override behavior.

## Decision

Use a signed HTTP-only cookie as the first-release area session mechanism and introduce a dedicated `/access` entry flow.

Operators will choose an area and submit that area's PIN through a dedicated access screen. The server will validate the PIN against environment-backed area configuration, issue a signed HTTP-only cookie containing the authenticated area identity and expiry metadata, and redirect the operator to the appropriate surface. The session will persist across routine requests during the shift and will be cleared only on explicit logout, area switch, or expiry.

Area PINs for `kitchen-1`, `kitchen-2`, and `salon` will be configured through environment variables rather than stored in SQLite for this release.

The cookie policy for this release is explicit:

- `HttpOnly`
- `Path=/`
- `SameSite=Lax`
- `Secure=true` outside development
- `Max-Age` derived from the configured shift-length TTL

The session uses a sliding renewal policy, but only near expiry. Protected page and API responses reissue the cookie only when the remaining lifetime falls below the renewal window, defined as the final 25% of the configured TTL. This avoids cookie churn on high-frequency polling routes while keeping fixed-device sessions alive during an active shift.

## Alternatives Considered

### Alternative 1: Client-Side Bearer Token

- **Description**: Issue a bearer token after PIN validation and store it in client-managed browser storage.
- **Pros**: Familiar API pattern and simple request decoration from the client.
- **Cons**: Exposes the session token to client-side JavaScript, increases accidental leakage risk on shared devices, and creates more client wiring than the current server-first routes need.
- **Why rejected**: The first release benefits more from server-owned session state than from client-managed token transport.

### Alternative 2: SQLite-Backed Area Credential Table

- **Description**: Store area PIN configuration in the local database and manage it through seed or direct database writes.
- **Pros**: Credentials become data rather than runtime config and could evolve toward an admin-managed model later.
- **Cons**: Adds schema, migration, seed, and operational update surface before there is a real need for runtime credential management.
- **Why rejected**: Environment-backed configuration is enough for the first release and avoids unnecessary persistence.

## Consequences

### Positive

- Keeps the session secret and signed payload on the server side.
- Works cleanly with Next.js server pages and route handlers.
- Avoids new database tables or admin UX in the first hardening slice.

### Negative

- PIN rotation requires environment updates and process restart or redeploy.
- Shared area sessions still provide area accountability, not individual accountability.

### Risks

- Missing or weak environment configuration could block login or reduce safety.
- Long-lived sessions on shared devices require a clear switch-area or logout path.
- Cookie signing mistakes could create hard-to-debug access failures.

## Implementation Notes

The dedicated access flow should support:

- initial login
- explicit area switch
- explicit logout
- session expiry handling

The first release should prefer a short, inspectable cookie payload with signed integrity rather than encrypted opaque session storage. The `next` redirect target must be validated against a per-area allowlist of canonical routes before any post-login redirect is honored.
