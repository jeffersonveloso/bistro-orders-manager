# ADR-002: Signal External Order Changes Without Rewriting the Kitchen Board

## Status

Accepted

## Date

2026-05-11

## Context

Once a confirmed external order has been imported into production, the provider may later report a change. That change could be harmless, important, or disruptive, depending on whether kitchen work has already started.

The product needs a clear rule for the first release. A fully automatic mirror of provider-side changes would reduce manual review in some cases, but it would also make the kitchen board less predictable during active production. The kitchens need confidence that once work starts, the board is not silently rewritten by an external system.

## Decision

In the first release, if an already imported order changes in the provider, the system will signal that an external change occurred without automatically rewriting the kitchen board.

The purpose of the signal is to:

- Preserve board predictability for the kitchen team
- Alert staff that the external order and the current board may now diverge
- Create a controlled exception path rather than hidden automatic changes

This decision applies even before deeper provider lifecycle coverage is added in later phases.

## Alternatives Considered

### Alternative 1: Auto-apply external changes only before kitchen work starts

- **Description**: Update the board automatically while the kitchen has not started the order.
- **Pros**: Reduces manual resolution for some edge cases.
- **Cons**: Introduces conditional behavior that is harder to explain and trust during rollout.
- **Why rejected**: The first release should favor a single predictable rule over clever partial automation.

### Alternative 2: Always mirror provider changes automatically

- **Description**: Keep the board in constant lockstep with the external provider.
- **Pros**: Maximum consistency with the provider surface.
- **Cons**: Can disrupt active production and make the board feel unstable under pressure.
- **Why rejected**: Kitchen trust and operational stability matter more than full external parity in the first release.

## Consequences

### Positive

- Preserves operator trust in the production board
- Makes external divergence visible instead of hidden
- Simplifies training with one consistent rule

### Negative

- Staff still need a manual resolution path for changed orders
- Some divergences may remain unresolved until a human intervenes

### Risks

- **Risk**: Alerting without a clear handling workflow could still create confusion.
  - **Mitigation**: Define an explicit exception-handling expectation in product copy and rollout training.
- **Risk**: Teams may overreact to low-impact external changes.
  - **Mitigation**: Keep the alert focused on meaningful order-change events only.

## Implementation Notes

- Treat the board as an operator-owned production surface after import.
- Design the external-change signal as an operational exception, not a passive log entry.
