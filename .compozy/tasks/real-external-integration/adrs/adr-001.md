# ADR-001: Controlled Confirmed-Order Ingestion for Real External Integration

## Status

Accepted

## Date

2026-05-11

## Context

The current product already runs a local kitchen production MVP with a mock provider adapter. The next product decision is how far the first real external integration should go.

The bistro's immediate need is to stop depending on manual order relay between the ordering platform and the two kitchens. The first release must create visible operational value for Kitchen 1 and Kitchen 2, which means new real orders need to appear on the production board without manual re-entry.

At the same time, this is the first production-facing integration. If the scope is too broad, the kitchen may lose trust in the board during live service. The team also wants explicit protection against duplicate imports and clear visibility when ingestion fails.

## Decision

The first release of the real external integration will focus on controlled ingestion of confirmed orders that are ready for production.

This scope includes:

- Automatic import of confirmed external orders into the production board
- Clear duplicate protection so the same external order does not create multiple production orders
- Clear visibility when an external order fails to enter the board
- A kitchen-first workflow where the board remains predictable once an order is in production

This scope excludes:

- Importing every upstream external state before confirmation
- Broad bidirectional synchronization as part of the first release
- Expanding into a general order-management surface for atendimento or management

## Alternatives Considered

### Alternative 1: Broader full-cycle external synchronization in the first release

- **Description**: Cover confirmed orders, upstream order changes, cancellations, and broad state synchronization from day one.
- **Pros**: More complete provider coverage and less future scope to add.
- **Cons**: Higher operational ambiguity, larger rollout risk, and more ways to break kitchen trust in the board.
- **Why rejected**: The first release should solve order entry reliability first, not every possible provider behavior.

### Alternative 2: Assisted live integration with heavy manual supervision

- **Description**: Use the real provider but keep the kitchen dependent on frequent manual checking and parallel processes.
- **Pros**: Smaller initial risk and lower expectation of automation.
- **Cons**: Delivers weaker operational improvement and prolongs duplicate work.
- **Why rejected**: The main business value is removing manual relay for new production orders.

## Consequences

### Positive

- Delivers immediate value to both kitchens by removing manual order repasse
- Keeps the first production rollout narrow enough to build operator trust
- Creates a clear acceptance bar for the release: confirmed orders must appear reliably in the board

### Negative

- Some external lifecycle events will remain outside the first release scope
- Atendimento and management users will not get a broad external order console yet

### Risks

- **Risk**: Stakeholders may expect full provider parity immediately.
  - **Mitigation**: Position the release as production-order ingestion, not complete provider synchronization.
- **Risk**: Operators may assume any missing order is a kitchen issue rather than an ingestion issue.
  - **Mitigation**: Make ingestion failures visible and operationally actionable.

## Implementation Notes

- Preserve the current kitchen-first operating model as the primary workflow.
- Treat duplicate protection and failure visibility as release-critical behavior, not polish.
