# ADR-004: Hybrid Ingestion With Webhook Trigger and Scheduled Reconciliation

## Status

Accepted

## Date

2026-05-11

## Context

The current application imports provider orders through a synchronous `OrderProviderPort` and serves read models through Next.js route handlers backed by SQLite. Phase 1 real integration must preserve the existing kitchen workflow while making confirmed provider orders appear quickly and reliably in the board.

The product requires near-real-time order entry, duplicate protection, failure visibility, and a recovery path when inbound events are lost or delayed. A single ingestion mechanism is unlikely to satisfy both latency and recoverability without overcomplicating the current architecture.

## Decision

Adopt a hybrid ingestion model:

- A provider-facing webhook route receives inbound order signals
- The webhook persists the inbound event and triggers canonical order synchronization
- A separate authenticated sync route is invoked by an external scheduler for periodic reconciliation
- Both paths flow through the same application-level sync pipeline

The webhook path is the fast path for Phase 1. Scheduled reconciliation is the safety net for lost webhooks, delayed provider delivery, and manual replays.

## Alternatives Considered

### Alternative 1: Polling-only ingestion

- **Description**: Rely only on scheduled provider polling to discover new or changed orders.
- **Pros**: Smaller public surface and simpler request authentication.
- **Cons**: Higher worst-case latency and weaker alignment with the PRD release gates.
- **Why rejected**: Phase 1 needs near-real-time intake for confirmed orders.

### Alternative 2: Webhook-only ingestion

- **Description**: Trust provider webhooks as the only input path.
- **Pros**: Lower sync frequency and simple happy-path flow.
- **Cons**: No built-in recovery when events are lost, duplicated, or delayed.
- **Why rejected**: Phase 1 requires explicit reconciliation and duplicate-resistant reliability.

## Consequences

### Positive

- Preserves low-latency intake for live kitchen operations
- Adds an explicit recovery path without changing kitchen-facing read models
- Keeps sync logic centralized instead of duplicating business rules across routes

### Negative

- Adds two authenticated write entry points instead of one
- Requires persisted sync bookkeeping to coordinate webhook and reconciliation paths

### Risks

- **Risk**: The scheduler may be misconfigured or stop running.
  - **Mitigation**: Emit sync-run health metrics and alert on stale reconciliation windows.
- **Risk**: Webhook and reconciliation paths may race on the same external order.
  - **Mitigation**: Use shared idempotency and transactional apply logic around canonical external order ids.

## Implementation Notes

- Keep webhook handlers thin; they should persist the event and invoke the shared sync application service.
- Reconciliation should use the same provider adapter and exception model as webhooks.
