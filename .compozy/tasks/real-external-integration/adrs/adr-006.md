# ADR-006: Fail Closed on Unmapped Provider Items

## Status

Accepted

## Date

2026-05-11

## Context

The current domain service rejects orders when any provider item lacks a local kitchen mapping. That behavior aligns with the MVP's explicit skip semantics but Phase 1 real integration needs a more structured operational outcome.

The PRD requires that orders with unmapped items stay out of the kitchen board until resolved. Partial imports or fallback kitchen routing would put incomplete or misleading work into production and break trust in the board.

## Decision

When a confirmed provider order contains any item without a local kitchen mapping, Phase 1 will reject the full import, record an explicit sync exception, and keep the order out of the kitchen board until the mapping issue is resolved and the order is replayed.

## Alternatives Considered

### Alternative 1: Partial import with unmapped item drop

- **Description**: Import mapped items and silently omit only the unknown items.
- **Pros**: Keeps some production moving.
- **Cons**: Produces incomplete kitchen work and hides a critical data problem.
- **Why rejected**: The board must not show a broken order as if it were production-ready.

### Alternative 2: Route unknown items to a fallback kitchen

- **Description**: Assign unmapped items to a default kitchen for manual triage.
- **Pros**: Avoids blocking intake.
- **Cons**: Creates operational ambiguity and offloads data-quality mistakes to kitchen staff.
- **Why rejected**: Kitchen operators should not absorb catalog-integration failures.

## Consequences

### Positive

- Preserves board trust and order completeness
- Makes mapping defects explicit and actionable
- Keeps the split logic deterministic

### Negative

- A single unmapped item blocks the full order from entering production
- Resolution depends on catalog maintenance outside the immediate intake path

### Risks

- **Risk**: A mapping miss during service could delay multiple orders.
  - **Mitigation**: Surface a clear exception and make replay easy after mapping is fixed.
- **Risk**: Stakeholders may view full-order rejection as too strict.
  - **Mitigation**: Document the trade-off as intentional protection for kitchen correctness.

## Implementation Notes

- Reuse the existing `MissingKitchenMappingError` as the domain signal, but persist an explicit exception record instead of console-only reporting.
- Support deterministic replay after the mapping is fixed.
