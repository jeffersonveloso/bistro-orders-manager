# ADR-005: Canonical Snapshot Sync With Dedicated SQLite Sync Tables

## Status

Accepted

## Date

2026-05-11

## Context

The current repository stores only internal production entities: orders, kitchen tickets, order items, kitchens, and menu item mappings. That is enough for mock imports, but not enough for real external integration where the system must explain what was received, what was applied, what failed, and why a given order now shows an exception marker.

The technical design needs durable sync state for webhook idempotency, reconciliation runs, and operator-visible exception history without mutating the current production aggregate for every provider-side change.

## Decision

Extend SQLite with dedicated sync and exception tables and treat the provider snapshot as the canonical sync input before applying internal business rules.

The design will include:

- persisted inbound provider events
- persisted sync run records
- persisted provider order snapshots or sync metadata sufficient to compare revisions
- persisted order-level sync exceptions linked to internal orders when present

Internal production tables remain the source of truth for kitchen execution. Sync tables explain how external information entered, failed, or diverged.

## Alternatives Considered

### Alternative 1: Reuse only existing production tables

- **Description**: Keep sync state implicit in `orders` and rely on logs for history.
- **Pros**: No schema expansion.
- **Cons**: Weak observability, no durable exception trail, and poor support for replay or reconciliation.
- **Why rejected**: Phase 1 explicitly requires visible failures and a minimal sync trail.

### Alternative 2: Single generic JSON audit table

- **Description**: Store all inbound events and outcomes in one flexible audit table.
- **Pros**: Fast initial implementation and low schema ceremony.
- **Cons**: Weak query ergonomics, muddled state ownership, and higher risk of implicit semantics buried in blobs.
- **Why rejected**: Phase 1 needs clear operational queries and explicit exception states.

## Consequences

### Positive

- Creates a durable audit trail for order intake and divergence
- Keeps the kitchen aggregate stable even when provider state changes
- Supports replay, reconciliation, and route-level diagnostics

### Negative

- Increases schema and migration scope
- Requires disciplined write ordering so sync tables and production tables stay coherent

### Risks

- **Risk**: Snapshot payload storage could grow faster than the production tables.
  - **Mitigation**: Store only the fields needed for sync decisions and debugging in Phase 1.
- **Risk**: Multiple tables may tempt over-modeling.
  - **Mitigation**: Limit Phase 1 to runs, events, provider order state, and exceptions only.

## Implementation Notes

- Use transactional writes so apply outcomes, events, and exceptions are recorded atomically.
- Keep provider-specific fields scoped to infrastructure and sync tables, not production read models.
