# ADR-008: Use Provider Catalog External IDs as the Canonical Menu Mapping Key

## Status

Accepted

## Date

2026-05-11

## Context

The current split logic depends on a stable `menuItemId` that maps directly to a local kitchen assignment. In the mock adapter this field is already normalized before the order reaches the domain.

For the real provider integration, item names are too unstable to serve as a safe mapping key. The Anota AI documentation also exposes catalog-level `externalID` flows, which creates a more reliable bridge between provider-side catalog items and local production mappings.

## Decision

The real Anota AI adapter will treat the provider catalog external ID as the canonical source for `menuItemId` in the internal normalized order input.

If a provider order item does not expose a usable catalog external ID that matches a local mapping key, the item is treated as unmapped and the order follows the fail-closed exception path defined in ADR-006.

## Alternatives Considered

### Alternative 1: Name-based matching

- **Description**: Match provider items to local mappings by normalized item name.
- **Pros**: Low upfront setup when catalog IDs are missing.
- **Cons**: Fragile under menu renames, accents, formatting, and duplicates.
- **Why rejected**: Kitchen routing must not depend on fuzzy text matching.

### Alternative 2: Separate local provider-id mapping table in Phase 1

- **Description**: Maintain a dedicated mapping from provider native item ids to local menu ids.
- **Pros**: Decouples local keys from provider catalog external IDs.
- **Cons**: Adds operational maintenance and another mapping layer before Phase 1 is proven.
- **Why rejected**: Provider external IDs already solve the mapping problem with less local complexity.

## Consequences

### Positive

- Keeps routing deterministic across provider and local systems
- Avoids name-based heuristics in production intake
- Fits the current split service contract with minimal translation logic

### Negative

- Requires provider catalog hygiene before live rollout
- Orders may block if the provider catalog is not populated with the expected external IDs

### Risks

- **Risk**: Catalog external IDs may be incomplete in the live provider account.
  - **Mitigation**: Validate the catalog before rollout and surface missing IDs as explicit exceptions.
- **Risk**: Teams may update local menu keys without coordinating provider catalog changes.
  - **Mitigation**: Document the mapping contract and treat key changes as controlled operations.

## Implementation Notes

- The adapter should extract the provider external ID before considering any weaker fallback.
- Phase 1 should not introduce fuzzy matching or alias lists.
