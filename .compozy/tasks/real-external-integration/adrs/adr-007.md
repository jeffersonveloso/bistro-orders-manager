# ADR-007: Shared Secret Authentication for Sync Entry Points

## Status

Accepted

## Date

2026-05-11

## Context

Phase 1 introduces provider-facing write endpoints that do not exist in the current MVP. These endpoints will either receive webhooks from Anota AI or be triggered by an external scheduler to run reconciliation. Both entry points can mutate local state and therefore need stronger protection than shape validation or passive logging.

The current repository has no authentication subsystem, and the PRD does not justify introducing a broader auth stack for internal kitchen users in this phase.

## Decision

Authenticate both webhook and scheduled sync routes with shared secret headers validated against environment variables on the server.

The webhook route and the scheduled reconciliation route may use separate secrets so they can be rotated independently and audited as distinct channels.

## Alternatives Considered

### Alternative 1: Accept unauthenticated requests with idempotency only

- **Description**: Rely on payload shape and duplicate protection to limit damage.
- **Pros**: Simplest handler implementation.
- **Cons**: Exposes state-changing routes to spoofed traffic and noisy abuse.
- **Why rejected**: Phase 1 write entry points must not trust the network implicitly.

### Alternative 2: IP allowlisting only

- **Description**: Restrict requests by source network without per-request shared secrets.
- **Pros**: Low application-level overhead.
- **Cons**: Harder to manage across environments and weaker when traffic paths change.
- **Why rejected**: Shared secrets are simpler to deploy consistently with the current stack.

## Consequences

### Positive

- Adds straightforward protection without introducing user auth complexity
- Keeps route authorization local to the integration surface
- Supports separate rotation policies for webhook and reconciliation callers

### Negative

- Requires operational secret distribution and rotation discipline
- Does not provide request signing or non-repudiation in Phase 1

### Risks

- **Risk**: Secret mismatches could block valid provider traffic.
  - **Mitigation**: Log authorization failures with route context and verify configuration in staging before pilot.
- **Risk**: Teams may overestimate the strength of simple shared secrets.
  - **Mitigation**: Keep the scope limited to Phase 1 and revisit stronger request validation later if needed.

## Implementation Notes

- Return `401` for missing or invalid secrets before any sync side effects occur.
- Keep header names explicit and documented in the TechSpec and README.
