# BUG-001: Auth test fixtures expired based on wall clock time

**Severity:** Medium
**Priority:** P1
**Type:** Functional
**Status:** Fixed

## Environment

- **Build:** local working tree during task 09 final verification
- **OS:** macOS
- **Browser:** N/A
- **URL:** `app/protected-pages.test.ts`, `app/api/catalog/routes.test.ts`

## Summary

The final QA rerun exposed a time-coupled test harness bug: several auth fixtures minted session cookies that expired at `2026-05-13T16:00:00.000Z`. Once the local clock crossed that instant, exported route and page tests started failing with `401` or `/access?reason=expired`, even though production behavior had not regressed.

## Reproduction

```bash
npm run test:run -- --coverage
```

Observed before the fix:

- `app/protected-pages.test.ts` failed while rendering authenticated pages because the mocked session was treated as expired.
- `app/api/catalog/routes.test.ts` failed on exported handlers because the request cookie was already expired when `withAreaSession()` used the real clock.

## Expected

Verification fixtures should be deterministic and must not depend on the local wall clock for passing or failing.

## Root cause

Multiple test helpers reused a hard-coded session expiry from the same calendar day as the QA run. Some suites injected `now` explicitly and stayed stable, but exported handlers and direct page renders used `new Date()` and therefore crossed the fixture expiry during the session.

## Fix

Updated the shared auth-fixture helpers in the affected test files to use a far-future expiry by default, while preserving explicit expired-session cases where the test intentionally controls the timestamp.

## Verification

- `npm run test:run -- app/api/catalog/routes.test.ts app/protected-pages.test.ts`
- `npm run lint`
- `npm run test:run -- --coverage`
- `npm run build`
- `npm run test:e2e`

## Impact

- **Users Affected:** QA and CI verification only
- **Frequency:** time-dependent, but guaranteed once the fixture expiry is crossed
- **Workaround:** rerun earlier in the day or inject `now` manually before the fix

## Automation Follow-up

- **Required:** No
- **Status:** Added
- **Spec / Command:** `app/protected-pages.test.ts`, `app/api/catalog/routes.test.ts`, related auth fixture helpers
- **Notes:** The fix removes the time-of-day flake without changing production auth logic.
